11. THE CORE PROBLEM (UNSOLVED)
================================
CVODE C loop = fast (11ms/pulse) but ne_eps goes negative.
scipy Python loop = physically correct but slow (1750ms/pulse).

The gap: per-step clamping of ne_eps and c_e.
- scipy does it in Python loop after each BDF step
- CVODE has no C-level hook for per-step user modification
- Python callback (CFUNCTYPE) for CVODE adds enough overhead to negate C loop benefit
- Numba @cfunc cannot capture closure (50+ array parameters)

Approaches tried and failed:
  a) CVODE CVodeSetConstraints: afterglow ne_eps decay too fast
  b) CVODE ONE_STEP + Python clamp: same speed as scipy (callback overhead)
  c) CVODE ONE_STEP + clamp + ReInit: BDF order reset kills dynamics
  d) asinh transform: ne_eps still goes negative (sinh allows negative)
  e) RHS structural guard (dydt=0 when ne_eps<floor): CVODE FD Jac ignores
  f) Analytical Jacobian: Newton diverges even with κ=10^6 scaling
  g) Semi-implicit Euler: order 1 insufficient, Te diverges
  h) Julia diffeqpy: Python RHS callback negates Julia solver speed


12. NEXT SESSION GOAL
=====================
Single goal: bridge the gap between CVODE speed and scipy correctness.

Most promising paths:
  a) Write RHS entirely in C (not Python/Numba) → CVODE calls C RHS directly
     → no callback overhead, can add clamping in C
  b) Modify SUNDIALS CVODE source to add per-step callback hook
  c) Use numbalsoda (Numba-compiled LSODA) — solver loop in compiled code
  d) Patankar scheme: production-destruction splitting guarantees positivity
     without external clamping

Key constraint: must reproduce baseline results exactly.
